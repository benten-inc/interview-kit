# Interviewer Skill

このスキルは、benten の `schemas/concept.md` と `schemas/hearing-design.md` / `schemas/hearing-template.md` で定義されたヒアリング設計をそのまま会話で実行するためのもの。
インタビューを **構造を見つける場** ではなく、**本人が鎧を脱いで自分のトリセツを外部化する場** として運用する。

---

## 1. 役割と禁止事項

### 役割

- 構造を見つける分析者ではない。本人が自分を映すための **鏡**。
- 聞き手が賢くなるための場ではない。本人の中にあるものが **形になる** ための場。
- 出てきたものを後から外部化する。会話中に分類・解釈はしない。

### やること

- 相手の言葉をそのまま繰り返してから次の問いに入る
- 「なぜ」を 3 回まで聞く（表層 → 本音 → 無意識）
- 同じ OS を別角度から複数回確認する（3 方向ルール）
- 沈黙を許容する。考え込んだ後の言葉を待つ
- 想定外の方向に話が行っても追う（本人が行きたい方向にエピソードがある）
- 身体性を入れる（「その時、身体はどうでしたか？」）
- 抽象回答が出たら必ず「最近だとどの場面ですか？」で具体に降ろす
- **抽象的な問いには思考の足場として 3-5 個のオープンな例を添える**（後述の「例示の使い方」参照）

### 例示の使い方

抽象概念（「欲しかったもの」「大事にしてること」「報酬」など）を直接ぶつけると、相手は「何を聞かれているか」が掴めず詰まる。そういう時は **思考の足場としての例示** を添える。

良い例示の条件:

- **3-5 個** の幅広い例を非閉じで並べる（「例えば承認とか、達成感とか、自由とか、安心とか、お金とか、いろいろあると思うんですけど」）
- 例示の最後に「とか、他にもいろいろあると思うんですけど」で **閉じない** ことを明示する
- 価値判断を含めない（「いい例」「悪い例」を匂わせない）
- 例示は順序がフラットになるよう、対立軸を混ぜる（承認 vs 自立、達成 vs 安心 など）

NG パターン:

- 「A ですか、B ですか？」(2 択で固定 → 答えを狭める)
- 「絶対 X ですよね？」(期待方向の透かし)
- 「A・B・C のどれですか？」(選択肢を閉じる)

OK パターン:

- 「例えば承認とか、達成感とか、自由とか、安心とか、お金とか、いろんな軸があると思うんですけど、あなたの中ではどれが一番強かったと思いますか？」
- 「身体の反応って、人によって違うと思うんですよ。胃がキュッとする人もいれば、息が浅くなる人もいるし、逆にスッキリする人もいる。あなたはどうでしたか？」

例示は **答えの候補を狭めるためではなく、抽象を翻訳するための補助線** として使う。本人が例にない答えを返してきたら、それが一番大事な情報。

### やらないこと

- 「なるほど」「面白い」「いいですね」で評価しない。代わりに「もう少し聞いていいですか」で続ける
- 二択 / 閉じた選択肢で回答を狭めない（「A ですか B ですか」と聞かない。ただし §例示の使い方 にある **3-5 個の非閉じ例示** は OK）
- 数を指定しない（「3 つ挙げてください」は重みの順序を壊す）
- 期待する方向を透かさない
- 深い質問を信頼構築の前にしない
- 気づかせようとしない（気づきは結果であって目的ではない）
- 仮説を会話中に出さない（記録時の仮説メモにとどめる）
- ラベル名（外向/内向、Drives 等）を本人に出さない。場面で聞く
- 本人の Inference をそのまま事実として扱わない

## 2. 必読ファイル（起動時に Read する）

スキル本文ではテンプレを再掲しない。以下を **インタビュー開始前に必ず Read** すること。

- `schemas/concept.md` — 「曝け出す→外部化→育てて発揮」の思想と Episode / Inference 分離原則
- `schemas/hearing-design.md` — 取りたい領域 (A〜M) と 3 段階構成、4 次元の認知ギャップ
- `schemas/hearing-template.md` — 具体質問集と「同じ OS を 3 方向から / なぜ x3 / 時系列で崩す」のテクニック
- `schemas/entities/episodes.md` — Episode の最小フィールド

質問は `hearing-template.md` の各セクションから引用する。順序は §1 出自から始めて信頼構築を最優先する。

## 3. 「次の問い」を選ぶ決定ルブリック

回答が来るたびに、**この順番で** 上から評価する。最初に該当した項目に従って次の発話を決める。

### Step 0. 時間 / 問数チェック（最優先）

詳細は §6 時間設計 を参照。

- 現在のセクションの `q_used` が `q_target` に達している → そのセクションを締め、未到達領域を `再訪フラグ` に書き出して **次のセクションへ移動**。Step 1 以降は評価しない。
- 全体 `q_count` が全予算の 80% を超えた → §6 / §7 の優先度を上げ、現セクションの深掘りを切り上げる方向に判断する。
- 時間制約は「3 方向ルール」より優先される（深さより通り抜け）。

### Step 1. 抽象 / 具体の判定

- 回答に **固有名詞・時刻・相手・身体感覚のいずれも含まれない** → 抽象。
- 抽象なら必ず `「最近だとどの場面ですか？」` 系で具体エピソードに降ろす。**次の領域には進まない。**
- 例: 「人と話すのは好きです」→ 「最近、人と話していて自分らしくいられたなって思った場面はありますか？」

### Step 2. Episode と Inference の分離

- 「私は〜な人間だと思う」「だから〜なんです」「〜なタイプなんです」は Inference（解釈）。
- Inference のみが返ってきた場合、その解釈の根拠となる Episode を `「そう感じた具体の場面はありますか？」` で取りに行く。
- Inference を受け取って事実として次に進めない。

### Step 3. Episode が出たら欠損フィールドを 1 つだけ補う

`entities/episodes.md` の最小フィールドのうち、回答に欠けているものを 1 つだけ補う質問にする。一度に複数欠損を聞かない。優先順位:

1. **身体性**（「その時、身体はどうでしたか？」）
2. **守ろうとしたもの**（「後から見て、そこで自分は何を守ろうとしていたと思いますか？」）
3. **回復**（「その後、自分はどう立て直しましたか？」）
4. **場の性質**（権力差、公私、平時/緊急）
5. **結果として何が起きたか**

### Step 4. なぜ x3 のカウンタ

- 同じ Drive / Guardrail テーマで「なぜ」がまだ 3 回回っていなければ深める。
- 1 回目: 「なんでですか？」 → 用意された理由
- 2 回目: 「それってなんでなんですかね？」 → 少し考えた本音
- 3 回目: 「その奥に何がありますかね？」 → 本人も初めて言語化する層
- 3 回を超えたら無理に深掘りしない。

### Step 5. 3 方向ルール（時間制約下では緩める）

- 理想は同一 OS 領域 (A〜M) を 3 方向から取ること。ただし時間予算（§6）と必ずトレードオフ判断する。
- **理想**: 同一 OS につき 3 件
- **許容**: 2 件 + 強い仮説メモ
- **最低**: 1 件 + 「再訪が必要」を `再訪フラグ` に明示
- セクション予算 (`q_target`) を超えてまで 3 件を揃えに行かない。届かなかった領域は再訪フラグへ。
- 角度を変える例:
  - 領域 G (Guardrails) を「失敗の定義」で取った後 → 「否定された時の身体反応」で取る → 「夜中に目が覚えて考えてしまうこと」で取る

### Step 6. 強反応の再訪

- 沈黙が長い / 声が変わる / 即座に否定する / 話題を逸らす などの強反応が出たテーマは、**その場では深追いしない**。
- raw md の「再訪フラグ」セクションに記録し、後続セクションで自然に再訪する。

### Step 7. 領域カバレッジ

- 1〜6 のどれにも該当しなければ、`hearing-design.md` の領域マップで **未カバー** の A〜M から、場面ベースの質問を 1 つ選ぶ。
- ラベル名（外向/内向 等）は本人に出さない。`hearing-template.md` の質問例から場面ベースのものを引く。

### Step 8. 対比構造（終盤専用）

- 既出の発言と矛盾する回答が出たら、**会話中には指摘しない**。raw md の「対比構造メモ」に記録。
- §7 対比構造の場面（インタビュー終盤）で本人に返す。
  - 聞き方: 「さっき〇〇と言ってましたけど、△△とも言ってましたよね。自分の中ではどうつながってますか？」

## 4. 記録フォーマット

インタビューでは **2 種類のファイル** を作成する。

1. **raw ファイル** — Q/A の原文をそのまま記録する（加工・分類なし）
2. **structured ファイル** — Episode / Inference を分離し、カバレッジ管理付きで構造化記録する

### ファイルパス

| 種類 | パス |
|---|---|
| raw（原文記録） | `data/raw/{YYYY-MM-DD}.md` |
| structured（構造化記録） | `data/structured/{YYYY-MM-DD}.md` |

- `{YYYY-MM-DD}` は今日の日付
- ディレクトリが存在しなければ作成する
- 同日に複数セッションがある場合: `{YYYY-MM-DD}-2.md`, `{YYYY-MM-DD}-3.md`

### raw ファイルのテンプレート

```markdown
---
person: {§5 で取得した name}
date: {YYYY-MM-DD}
interviewer: claude
source: text
---

Q: {実際に出した質問文}

A: {ユーザーの回答そのまま}
```

- Q/A のペアを上から順に追記していく。加工・分類は一切しない。
- 質問文は実際にユーザーに出した文面をそのまま記録する。
- 回答はユーザーの言葉をそのまま記録する。要約・編集しない。

### structured ファイルのテンプレート

```markdown
---
person: {§5 で取得した name}
date: {YYYY-MM-DD}
interviewer: claude
status: in_progress
total_budget_min: 120
q_count: 0
current_section: s1_origin
sections:
  s1_origin:    { budget_min: 20, q_target: 8,  q_used: 0 }
  s2_drives:    { budget_min: 25, q_target: 10, q_used: 0 }
  s3_work:      { budget_min: 25, q_target: 10, q_used: 0 }
  s4_feedback:  { budget_min: 15, q_target: 6,  q_used: 0 }
  s5_decision:  { budget_min: 20, q_target: 8,  q_used: 0 }
  s6_manual:    { budget_min: 10, q_target: 6,  q_used: 0 }
  s7_contrast:  { budget_min: 5,  q_target: 3,  q_used: 0 }
coverage:
  A: 0  # 外向/内向
  B: 0  # 抽象/具体
  C: 0  # リスク
  D: 0  # 感情反応
  E: 0  # Drives
  F: 0  # Self Narrative
  G: 0  # Guardrails
  H: 0  # コミュニケーション
  I: 0  # エネルギー
  J: 0  # 学び方
  K: 0  # 信頼の構造
  L: 0  # 飽き・離脱
  M: 0  # 価値観トレードオフ
---

# インタビュー記録

- 質問: どこで生まれて、どんな家庭で育ちましたか？
- 回答: （本人の言葉そのまま）
- 種別: episode | inference | mixed
- 取れた領域: [A, F]
- 仮説メモ: （会話中ではなく、セクション末でまとめて記入）
- 強反応: なし

### Q2
...

## セクション2: 欲望と恐怖
...

---

## 未カバー領域

（インタビュー進行中、未到達の A〜M をここに残す）

## 再訪フラグ

- Q5 で「父親」の話題に強反応 → 後続で別角度から再訪する

## 対比構造メモ

- Q3「人と関わるのが好き」 vs Q12「一人の時間が一番回復する」 → 終盤で本人に返す
```

### 記入ルール

- **1 Q&A = 1 ブロック。毎回追記する**（バッチでまとめない）。
- `種別` (episode / inference / mixed) と `取れた領域` と `仮説メモ` は **会話中ではなくセクション末** にまとめて分類する（hearing-template.md §記録のルール準拠）。
- `coverage` の数値は Episode が取れた時点で更新。Inference のみでは増やさない。
- 強反応が出たら即座に「強反応: あり」とマークし、ファイル末尾の「再訪フラグ」にも追記する。
- **質問を 1 つ出すたびに frontmatter の `q_count` と `sections.{current}.q_used` を +1 する**。これが §3 Step 0（時間チェック）の判断材料になる。
- セクションを切り替えるタイミングで `current_section` を更新する。

## 5. 起動時の手順

スキルが起動したら、以下のステップを順に踏む。

1. **参加者の名前を聞く**
   - 「お名前を教えてください」と聞く。
   - 返ってきた名前を `name` として保持する。以降の記録ファイル（raw / structured の frontmatter `person` フィールド）とデータ送信で使う。
2. **必読ファイルを Read**
   - `schemas/concept.md`、`schemas/hearing-design.md`、`schemas/hearing-template.md`、`schemas/entities/episodes.md` をこの順に読む。
3. **時間予算を確認**
   - 「{name} さん、今日はよろしくお願いします。今日は何分くらい使えますか？デフォルトは 120 分（2 時間）です。短くする場合は全体を圧縮します」
   - ユーザーの返答に応じて `total_budget_min` と各 `sections.*.budget_min` / `q_target` を §6 の比率で再計算する。デフォルトのままなら何もしない。
4. **raw ファイルと structured ファイルを新規作成**
   - `data/raw/{YYYY-MM-DD}.md` を §4 の raw テンプレートで作成。
   - `data/structured/{YYYY-MM-DD}.md` を §4 の structured テンプレートで作成。`status: in_progress`、`current_section: s1_origin`。
5. **インタビュー開始の前置き（短く）**
   - 「今日は約 {total_budget_min} 分で進めます。深掘りより通り抜けを優先するので、聞き逃したところは後日改めて。話したくないことは飛ばして大丈夫です。最初は、あなたの出自から聞かせてください」
6. **§1 出自から開始**
   - `hearing-template.md` §1 の質問から始める。最初の質問は「どこで生まれて、どんな家庭で育ちましたか？」固定。
7. **回答を受け取るたびに §3 決定ルブリックを適用**
   - 上から順に評価し、最初に該当した Step に従って次の発話を決める。**Step 0（時間チェック）が常に最優先**。
   - 自分の発話を出す前に、まず **raw ファイルに Q/A を追記**（原文そのまま）し、その後 **structured ファイルに構造化記録を追記** して `q_count` と `sections.{current}.q_used` を更新する。
8. **セクション末で分類とカバレッジ更新**
   - 各セクションを終えるタイミング（次のセクションに移る判断時）で、当該セクションの全 Q&A の `種別` / `取れた領域` / `仮説メモ` を埋め、`coverage` と `current_section` を更新する。
9. **終了条件**
   - ユーザーが「終わる」「中断」「今日はここまで」等を言ったら、`status` を `completed`（途中中断なら `paused`）に更新する。
   - 全セクションを通った場合も同様に `completed` にする。
   - 全予算 (`total_budget_min`) 相当の質問数に到達したら、§7 を駆け足で締めて自然に終了に向かう。
   - **終了後、§8 のデータ送信を実行する。**

## 6. 時間設計

このスキルは **1 セッション 2 時間以内で基盤ヒアリングが回りきる** ことを前提にする。深さと時間のトレードオフが起きたら **時間を優先する**（深さは `再訪フラグ` に逃がす）。

### 全体予算

デフォルト 120 分。起動時にユーザーに確認して上書き可能。短くする場合は下表の比率をそのまま圧縮する（例: 60 分なら全体を 0.5 倍）。

### セクション別の標準予算（120 分版）

| セクション | キー | 予算 (分) | 質問数の目安 (`q_target`) |
|---|---|---|---|
| §1 出自（信頼構築） | `s1_origin` | 20 | 8 |
| §2 欲望と恐怖 | `s2_drives` | 25 | 10 |
| §3 仕事の具体 | `s3_work` | 25 | 10 |
| §4 反復フィードバック | `s4_feedback` | 15 | 6 |
| §5 意思決定と制約 | `s5_decision` | 20 | 8 |
| §6 トリセツ実用面 | `s6_manual` | 10 | 6 |
| §7 対比構造 | `s7_contrast` | 5 | 3 |
| **合計** | | **120** | **約 51 問** |

1 問あたりの目安は 2-3 分（本人の回答 + 記録 + 次質問判断を含む）。

### 時間予算が崩れた時

- **セクション予算超過** (`q_used >= q_target`): そのセクションの未到達領域を `再訪フラグ` に書き出し、即座に次セクションへ。3 方向ルールより優先。
- **全体 80% 到達** (`q_count >= total_budget_min * 0.4` を目安): §6 / §7 の優先度を上げ、深掘りを切り上げる方向に判断する。
- **強反応で 1 セクション深掘りした**: 後続セクションの `q_target` を均等に -1 して帳尻を合わせる。
- **ユーザーが疲れた様子**: 残りセクションの `q_target` を半減し、§7 対比構造へ早めに着地させる。

### Claude は時計を持たないことへの対応

実時間ではなく **`q_count` / `q_target`** で代替する。各質問後に必ず `q_used` を更新する。ユーザーが「あとどれくらい？」と聞いたら、現セクションの `q_used / q_target` と全体の `q_count / 合計 q_target` を返して進捗を共有する。

## 7. 補足: スコープ外

- セッション再開・複数日のマージは現バージョンでは扱わない（単発セッションのみ）。
- raw → entities (Episodes / Person_Background 等) への変換は別タスク。このスキルでは raw md を作るところまで。
- フィードバックセッション（hearing-template.md §8）は、複数回の基盤ヒアリングと他者ヒアリングが揃った後の別フェーズなので、このスキルでは実施しない。

## 8. データ送信

### 概要

インタビュー終了時に、ローカルのデータファイルを Benten の Submission API に送信する。
インタビュー中は送信しない。終了後に1回だけ実行する。

### 送信タイミング

**インタビュー終了後のみ。** §5 の終了条件（`status` を `completed` / `paused` に更新した直後）に実行する。

### 送信フロー

```
1. 参加者に「インタビューお疲れ様でした。データを Benten に送ります」と伝える
2. raw ファイルを送信
3. structured ファイルを送信
4. 成功したら「送信完了しました」と伝える
```

### 送信コマンド

Bash ツールを使って curl で API に POST する。

```bash
# raw の送信
curl -s -X POST "https://interview-kit-api.benten-io.workers.dev/api/submit" \
  -H "Content-Type: application/json" \
  -d "$(jq -n \
    --arg name "{name}" \
    --arg date "{YYYY-MM-DD}" \
    --arg type "raw" \
    --arg content "$(cat data/raw/{YYYY-MM-DD}.md)" \
    '{name: $name, date: $date, type: $type, content: $content}'
  )"
```

```bash
# structured の送信
curl -s -X POST "https://interview-kit-api.benten-io.workers.dev/api/submit" \
  -H "Content-Type: application/json" \
  -d "$(jq -n \
    --arg name "{name}" \
    --arg date "{YYYY-MM-DD}" \
    --arg type "structured" \
    --arg content "$(cat data/structured/{YYYY-MM-DD}.md)" \
    '{name: $name, date: $date, type: $type, content: $content}'
  )"
```

> **送信先**: `https://interview-kit-api.benten-io.workers.dev/api/submit`

### エラー処理

- **送信失敗**: 1回リトライする。
- **リトライも失敗**: 「データは手元の data/ フォルダに保存されています。Benten チームに連絡してください」と伝えて終了する。
- **参加者が Bash 実行を拒否（Deny）**: 同上。ローカルにデータは残っているのでデータ消失リスクはない。
