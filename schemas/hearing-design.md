# ヒアリング設計

## 目的

その人の中にあるものを、安全な場で引き出し、外部化するための設計。

分析のためにデータを集めるのではない。
本人が鎧を脱いで話せる場を作り、出てきたものを形にして残す。

3段階 + フィードバックで行う：

1. **基盤ヒアリング** — Person_Background と Person_Experience の仮説を立てる
2. **認知ギャップヒアリング** — 本人と周囲のズレを読むための Episodes を集める
3. **フィードバックセッション** — 集めた構造を本人に返す。詳細は `concept.md` のフィードバック設計を参照

---

## 1. 基盤ヒアリング

### 見たい領域

#### Person_Background（比較的安定した傾向）

| ID | 領域 | 見たいこと | 聞き方の例 |
|----|------|-----------|----------|
| A | 外向 / 内向 | 社交がOSかskillか | 人と長く会った後、どう過ごしたくなるか。人前でうまく振る舞える時、自然か頑張っているか |
| B | 抽象 / 具体 | 構造で掴むか手触りで掴むか | 何かを説明する時、全体から入るか具体から入るか。うまくいった理由を構造で考えるか出来事で考えるか |
| C | リスク | 不確実さが燃料か安定が土台か | 先が見えない仕事でワクワクするか身構えるか。現状維持に違和感があるか |
| D | 感情反応 | どれくらい揺れるか、残るか | 否定された時どれくらい引きずるか。身体に出るか。切り替えは早いか |
| H | コミュニケーション | 情報をどう処理し、どう伝え、どう受け取るか | 考えがまとまる前に話し始めるか、考えてから話すか。大事なことはテキストと口頭どちらが入りやすいか |
| I | エネルギー | どういうリズムで動き、何で充電し、何で壊れるか | 1日の中で一番冴えている時間帯。人以外で意外と消耗するもの。限界の前兆 |
| J | 学び方 | 未知のものをどう掴み、どう身につけるか | 新しいことを始める時まず何をするか。全体像が先か手を動かすのが先か |

#### Person_Experience（行動を駆動するもの）

| ID | 領域 | 見たいこと | 聞き方の例 |
|----|------|-----------|----------|
| E | Drives | 何を報酬として感じるか | 「報われた」と感じた瞬間。誰にも言われなくても追いかけてしまうもの。気が済まないこと |
| F | Self Narrative | どんな自己像を守っているか | どう思われたいか。絶対に自分はそうではないと思うもの。周りからどう見られると一番しっくりくるか |
| G | Guardrails | 何を危険と感じるか | 何を否定されると一番きついか。急に身構える場面。追い込まれた時に何を後回しにするか |
| K | 信頼の構造 | 何で信頼し、何で壊れ、戻るか | 人を信頼する時何を見ているか。一発で信頼がなくなるのはどんな時か |
| L | 飽き・離脱 | 何でエンジンが止まるか | 続かなかったものの共通点。不思議なくらい続いているもの |
| M | 価値観トレードオフ | 大事なもの同士がぶつかった時どちらを取るか | 正しさと調和で迷った場面。効率と人の気持ちがぶつかった場面 |

### 進め方

- ラベルを直接聞かない。具体的な場面で聞く
- 「できるか」ではなく「自然にそうなるか」
- 「好きか」ではなく「エネルギーがどう動くか」
- 回答が抽象的なら「最近だとどの場面ですか」で具体に戻す
- 相手の言葉をそのまま事実にしない（Episode と Inference を分離する）

### 残すもの

- 仮説メモ（結論ではなく仮説）
- 後で検証したいポイント
- 繰り返し語ったキーワード
- 強い反応が出たテーマ

## 2. 認知ギャップヒアリング

### 認知ギャップとは何か

単なる「誤解」ではない。**同じ事象に対して、異なる次元で解釈している状態**。
どちらかが間違っているのではなく、処理している次元が違う。

ズレが生まれる次元は4つ：

| 次元 | 定義 | 例 |
|------|------|-----|
| **価値観** | 何を大事にしているかが違う | 本人は事実/効率を優先 → 相手は関係性/安全を優先 |
| **抽象度** | どの粒度で処理しているかが違う | 本人は構造の話 → 相手は「来月何するの？」 |
| **前提** | 当たり前にしている暗黙知が共有されていない | 本人は「リソースは作るもの」→ 相手は「有限な制約」 |
| **視点の高さ** | 見ている時間軸・スコープが違う | 本人は3年後を見ている → 相手は今期を見ている |

### 認知ギャップのパターン

各パターンに4次元のどれが主因かを併記。

| パターン | 本人の認識 | 相手の受け取り | 主な次元 |
|---------|----------|-------------|---------|
| 威圧 / 率直 | 正しいことを早く言った | 攻撃された | 価値観（事実 vs 安全） |
| 閉じる / 拒絶 | 考えている | 怒っている、拒絶 | 価値観（思考 vs 感情） |
| 任せる / 放置 | 信頼して任せた | 見捨てられた | 価値観（成長 vs 安全） |
| 基準 / 冷たさ | 組織として正しい判断 | 人を見ていない | 価値観（構造 vs 承認） |
| 外向skill / 内向OS | この場では機能した | この人はこういう人だ | 価値観（機能 vs 本質） |
| 有事 / 平時 | 追い込まれて反応が変わった | この人の本性が出た | 価値観（状況適応 vs 一貫性） |
| 事業選択 / 自己投影 | 合理的に選んだ | 個人の物語が乗りすぎ | 前提（本人も無自覚な動機） |
| 意思決定の速さ | 速く決めた | 粗い、恣意的 | 前提 + 抽象度 |
| 方針変更 / ブレ | 学習して更新した | 一貫性がない | 視点の高さ + 前提 |
| リスク / 無謀 | 勝ち筋を見て張った | 無謀 | 視点の高さ + 前提 |
| 長期投資 / 遅さ | 長期の構造を見ている | 今を見ていない | 視点の高さ |
| 評価 / 冷たさ | 実力で評価した | 人を見ていない | 価値観 + 抽象度 |

### 聞き方（4次元共通）

**価値観のズレを取る：**
- 「その時、あなたが一番大事にしてたことは？」
- 「相手が大事にしていたものって、同じだった？」

**抽象度のズレを取る：**
- 「その話、相手にはどこまで伝わっていたと思う？」
- 「自分が見えていて、相手に見えていなかったものは？」

**前提のズレを取る：**
- 「それって、言わなくても伝わると思ってた？」
- 「あなたの当たり前で、相手の当たり前じゃなかったものは？」

**視点の高さのズレを取る：**
- 「その判断、何年先を見てた？」
- 「相手はどの時間軸で見てたと思う？」

### 優先して聞く場面

- 部下や共同創業者とのズレ
- 採用・初対面での印象差
- 投資家・顧客との誤解
- 場によって印象が変わった場面
- 圧がかかって判断や言動が変わった場面
- 任せたつもりが伝わらなかった場面
- 正しさを優先して関係に歪みが出た場面

## 3. Episode の取り方

Episode のフィールド定義は `concept.md` の Episodes を参照。
具体的な聞き方は `hearing-template.md` を参照。

ここでは設計上の原則だけ記載する：

- 基本フォーマット（何が起きたか、相手、場の性質、行動、感覚、結果、回復、守ろうとしたもの）は毎回取る
- 認知ギャップ構造（4次元）、対比構造、意思決定の軸は、該当する Episode に付与する
- すべてのフィールドを毎回埋める必要はない。会話の流れで自然に出たものを記録する

## 4. 他者ヒアリング

### 原則

欲しいのは性格評価ではなく、**その人がどう見えたか、その印象がどの場面から来ているか**。

### 聞く順番

1. 何が起きたか
2. どう受け取ったか
3. どう感じたか
4. その受け取りはどの場面から来ているか
5. 例外はあったか

### 取るもの

| フィールド | 意味 |
|-----------|------|
| perceived_pattern | どんな人に見えているか |
| perceived_gap | 何がズレとして感じられているか |
| context_scope | その像はどの文脈で強く出るか |
| episode_count | いくつの場面に基づくか |
| bias_risk | 関係や感情による歪みの可能性 |

### 質問例

- あなたから見ると、その人はどんな人に見えますか
- その印象を強く持った具体的な場面は？
- その人と仕事していて、安心する時と身構える時は？
- その人が黙った時、どう解釈しますか
- 普段の印象と違った場面はありましたか
- 振り返ると、自分が誤解していたかもしれないと思う場面は？

---

## 5. 聞き手の振る舞い

- ラベルを直接聞かない。場面で聞く
- 「なぜ」を3回聞く（表層→本音→無意識）
- 同じOSを3方向から確認する
- 沈黙を許容する
- 想定外の方向に行っても追う
- Inference（本人の解釈）を事実にしない
- 仮説は会話中に出さない
- 「最近だとどの場面？」で具体に戻す
- 強く反応したテーマは後続で必ず再確認する

## 6. フィードバックセッション

フィードバックの思想・設計は `concept.md` のフィードバック設計と活用設計を参照。
具体的な質問は `hearing-template.md` のセクション8を参照。

ここでは設計上の位置づけだけ記載する：

- 基盤ヒアリング + 認知ギャップヒアリング + 他者ヒアリングが完了した後に行う
- Validation を通過した構造がある場合にのみ実施する
- 目的は「伝える」だけでなく「活用できる形にする」まで含む
