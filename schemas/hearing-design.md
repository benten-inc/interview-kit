# ヒアリング設計

## 目的

その人の中にあるものを、安全な場で引き出し、外部化するための設計。

分析のためにデータを集めるのではない。
本人が鎧を脱いで話せる場を作り、出てきたものを形にして残す。

## フローの構造

このモデルはステージ型のフローを取らない。
「基盤ヒアリング → 認知ギャップヒアリング → 他者ヒアリング → フィードバック」という固定順序ではなく、
**状態マップが「次に何を聞くべきか」「誰に聞くべきか」を常に判断する**ループを回す。

```
┌─→ 状態マップ確認
│     ↓
│   判断ルブリック: what × who
│     ↓
│   ┌── best_source: self ──→ 本人セッション
│   └── best_source: other ─→ 他者セッション（対象者指定）
│     ↓
│   セッション後分析（状態マップ更新）
│     ↓
│   収束条件チェック → 満たした → フィードバック → 育成ループ
│     ↓ まだ
└─────┘
```

初回は状態マップが全領域空白のため、必ず本人から始まる。
回を重ねると、深掘り・矛盾検証・他者ヒアリングへ自然に移行する。

## 2. 本人セッションの設計

### セッション冒頭のプロトコル（毎回固定）

初回以外のすべての本人セッションは、以下から始める。

1. 「前回の後、考えたこととか気になったことってありましたか？」
2. 出てきたものをまず全部聞く。遮らない
3. 何も考えなかった場合、それ自体が材料（前回のテーマがその人にとって軽かった可能性）
4. その後、判断ルブリックに従って本題に入る

### 判断ルブリック: what（何を聞くか）

セッション冒頭で状態マップを見て、以下の優先順位で判断する。

| 優先度 | 条件 | やること |
|--------|------|---------|
| 1 | 毎回 | 前回から考えたことを聞く |
| 2 | unresolved_contradictions に priority: high がある | 例外質問 or 別角度の場面で矛盾を浮かせる |
| 3 | confidence: low の領域がある | その領域に触れる場面を聞く |
| 4 | confidence: medium の領域がある | 別文脈の Episode を取る / 有事で確認する |
| 5 | 上記すべてが一定水準 | 新しいエピソードを自由に聞く |

### 信頼レベルと質問の深さ

ループ型では「何回目だからこの質問」ではなく、信頼レベルで聞ける深さが決まる。

| trust_level | 聞ける範囲 | 聞けない範囲 |
|-------------|----------|------------|
| low | 出自、仕事の具体、得意なこと | 恐怖、Guardrails の核心、性格の矛盾 |
| medium | 欲望、自己像、意思決定の癖 | 最も深い不安、誰にも言えないこと、矛盾の直接的な浮上 |
| high | 有事の反応、矛盾、限界、死角 | なし（ただし伝え方は常に慎重に） |

trust_level の判定基準:

| レベル | 根拠の例 |
|--------|---------|
| low → medium | 感情を伴うエピソードを自発的に語った。弱さに関する話題を避けなかった |
| medium → high | 前回のセッション後に自分から追加の話をしてきた。「実は…」で始まる話が出た。矛盾を指摘されても防衛が上がらなかった |

### 言語化の質の追跡

きれいに言語化された回答ほど、再確認が必要。

| verbalization_quality | 意味 | 次のアクション |
|----------------------|------|--------------|
| high（きれいにまとまっている） | 本人の言語能力で整えられた可能性 | 別角度から再確認、場面ベースで具体に降ろす、身体感覚を聞く |
| medium（部分的に言語化できている） | 一部は構造に近い | 言語化できていない部分を場面で埋める |
| low（言いよどんでいる、修正が多い） | 言葉になる前の構造に近い可能性 | 急かさない。沈黙を許容。言い直しの方向を記録する |

### 進め方の原則

- ラベルを直接聞かない。具体的な場面で聞く
- 「できるか」ではなく「自然にそうなるか」
- 「好きか」ではなく「エネルギーがどう動くか」
- 回答が抽象的なら「最近だとどの場面ですか」で具体に戻す
- 相手の言葉をそのまま事実にしない（Episode と Inference を分離する）
- きれいにまとまった回答には「今の言葉、自分的にしっくり来てますか？」で精度を確認する

## 3. 他者セッションの設計

### いつ他者に聞くか

他者ヒアリングは独立したフェーズではない。
状態マップの open_questions に `best_source: other` が出た時に走る。

#### best_source: other になる問い

| 問いの性質 | 例 |
|-----------|-----|
| 本人の行動が他者にどう見えているか | 「率直さが威圧に見えているか」 |
| Self Narrative の外部検証 | 「"人を大事にしている" は周囲からも見えているか」 |
| 認知ギャップの特定 | 「本人の沈黙をどう受け取っているか」 |
| 本人が語れない死角 | 「本人が気づいていない影響は何か」 |

### 他者セッションの構造

```
1. 自由印象（5-10分）
   - 汎用的に語ってもらう。予想外の材料が出る場所
   - 「〇〇さんと仕事していて、どんな印象ですか？」

2. 焦点化（本体）
   - 状態マップの open_question に沿って具体場面を聞く
   - 「〇〇さんに大事にされてるなって感じた場面と、
     逆にちょっと冷たいなって感じた場面、両方あれば聞かせてもらえますか？」

3. 例外確認
   - 「普段の印象と違った場面はありましたか？」
   - 「振り返ると、自分が誤解していたかもしれないと思う場面は？」
```

### 聞く順番

1. 何が起きたか
2. どう受け取ったか
3. どう感じたか
4. その受け取りはどの場面から来ているか
5. 例外はあったか

### 取るもの

| フィールド | 意味 |
|-----------|------|
| perceived_pattern | どんな人に見えているか |
| perceived_gap | 何がズレとして感じられているか |
| context_scope | その像はどの文脈で強く出るか |
| episode_count | いくつの場面に基づくか |
| bias_risk | 関係や感情による歪みの可能性 |
| triggering_question | 状態マップのどの問いに対応するか |
| state_map_field | 状態マップのどの領域に効くか |

### 原則

欲しいのは性格評価ではなく、**その人がどう見えたか、その印象がどの場面から来ているか**。

## 4. 認知ギャップの取り方

### 認知ギャップとは何か

単なる「誤解」ではない。**同じ事象に対して、異なる次元で解釈している状態**。
どちらかが間違っているのではなく、処理している次元が違う。

ズレが生まれる次元は4つ：

| 次元 | 定義 | 例 |
|------|------|-----|
| **価値観** | 何を大事にしているかが違う | 本人は事実/効率を優先 → 相手は関係性/安全を優先 |
| **抽象度** | どの粒度で処理しているかが違う | 本人は構造の話 → 相手は「来月何するの？」 |
| **前提** | 当たり前にしている暗黙知が共有されていない | 本人は「リソースは作るもの」→ 相手は「有限な制約」 |
| **視点の高さ** | 見ている時間軸・スコープが違う | 本人は3年後を見ている → 相手は今期を見ている |

### 認知ギャップのパターン

各パターンに4次元のどれが主因かを併記。

| パターン | 本人の認識 | 相手の受け取り | 主な次元 |
|---------|----------|-------------|---------|
| 威圧 / 率直 | 正しいことを早く言った | 攻撃された | 価値観（事実 vs 安全） |
| 閉じる / 拒絶 | 考えている | 怒っている、拒絶 | 価値観（思考 vs 感情） |
| 任せる / 放置 | 信頼して任せた | 見捨てられた | 価値観（成長 vs 安全） |
| 基準 / 冷たさ | 組織として正しい判断 | 人を見ていない | 価値観（構造 vs 承認） |
| 外向skill / 内向OS | この場では機能した | この人はこういう人だ | 価値観（機能 vs 本質） |
| 有事 / 平時 | 追い込まれて反応が変わった | この人の本性が出た | 価値観（状況適応 vs 一貫性） |
| 事業選択 / 自己投影 | 合理的に選んだ | 個人の物語が乗りすぎ | 前提（本人も無自覚な動機） |
| 意思決定の速さ | 速く決めた | 粗い、恣意的 | 前提 + 抽象度 |
| 方針変更 / ブレ | 学習して更新した | 一貫性がない | 視点の高さ + 前提 |
| リスク / 無謀 | 勝ち筋を見て張った | 無謀 | 視点の高さ + 前提 |
| 長期投資 / 遅さ | 長期の構造を見ている | 今を見ていない | 視点の高さ |
| 評価 / 冷たさ | 実力で評価した | 人を見ていない | 価値観 + 抽象度 |

### 聞き方（4次元共通）

**価値観のズレを取る：**
- 「その時、あなたが一番大事にしてたことは？」
- 「相手が大事にしていたものって、同じだった？」

**抽象度のズレを取る：**
- 「その話、相手にはどこまで伝わっていたと思う？」
- 「自分が見えていて、相手に見えていなかったものは？」

**前提のズレを取る：**
- 「それって、言わなくても伝わると思ってた？」
- 「あなたの当たり前で、相手の当たり前じゃなかったものは？」

**視点の高さのズレを取る：**
- 「その判断、何年先を見てた？」
- 「相手はどの時間軸で見てたと思う？」

### 優先して聞く場面

- 部下や共同創業者とのズレ
- 採用・初対面での印象差
- 投資家・顧客との誤解
- 場によって印象が変わった場面
- 圧がかかって判断や言動が変わった場面
- 任せたつもりが伝わらなかった場面
- 正しさを優先して関係に歪みが出た場面

## 5. Episode の取り方

Episode のフィールド定義は `concept.md` の Episodes を参照。
具体的な聞き方は `hearing-template.md` を参照。

ここでは設計上の原則だけ記載する：

- 基本フォーマット（何が起きたか、相手、場の性質、行動、感覚、結果、回復、守ろうとしたもの）は毎回取る
- 認知ギャップ構造（4次元）、対比構造、意思決定の軸は、該当する Episode に付与する
- すべてのフィールドを毎回埋める必要はない。会話の流れで自然に出たものを記録する
- セッション追跡フィールド（session_date, session_number, is_revisit, revisit_of）を必ず付与する

## 6. セッション後分析

**すべてのセッション後に実行する。** ループの中核。

### 手順

```
1. Episode の構造化
   - 今回取れた Episode を構造化する
   - 事実（Episode）と本人の解釈（Inference）を分離する
   - 言語化の質を記録する（きれいすぎないか、言いよどみはあったか）

2. Inferences の生成
   - Episode 単位の Inference を生成する
   - 前回までの Episode との突合から、セッション横断の Inference を生成する
     - 同じ領域で一貫しているか
     - 前回と矛盾する材料が出たか
     - 前回きれいに言語化されていたものが、今回は別の形で現れたか

3. 状態マップの更新
   - 各領域の confidence / depth を更新する
   - 新しい self_narrative_claim があれば追加する
   - 既存の self_narrative_claim に対して、支持/矛盾する Episode を紐づける
   - open_questions の追加・解決・best_source 判定
   - trust_level の更新
   - verbalization_quality の更新

4. Self_Narrative_Divergence の候補検出
   - Self Narrative の主張と、今回の Episode を突合する
   - 乖離候補があれば unresolved_contradictions に追加する
   - 既存の乖離候補に対して、追加の証拠があれば紐づける

5. Validation（十分な材料が揃った領域のみ）
   - 複数文脈で再現しているか
   - 時間差をおいて同じ構造が出たか
   - 反証がないか
   - 採択 / 保留 / 棄却 を判定する

6. 次回セッションの設計
   - 判断ルブリックで優先順位を決める
   - best_source を判定する（次は本人か他者か）
   - best_source: other の場合、対象者を特定する

7. 収束条件のチェック
   - Person_Pattern の主要領域が confidence: high か
   - Self_Narrative_Divergence が少なくとも1つ validated か
   - open_questions に priority: high がないか
   - trust_level が high か
```

### diminishing returns の検出

同じ領域を3回以上触って confidence が上がらない場合:

1. 聞き方を変える（場面ベース → 身体感覚 → 他者視点）
2. それでも動かなければ「この領域は現時点ではここまで」と保留にする
3. best_source を other に切り替えて、外から取りに行く

## 7. 矛盾の扱い方

矛盾を指摘すると防衛が上がり、場が壊れる。
矛盾は「指摘する」のではなく「自然に浮かぶ場面を作る」。

### テクニック

**両方を肯定して並べる：**
> 「〇〇の話も分かるし、△△の話もすごく分かるんですよ。これって自分の中ではどういう感じですか？」

矛盾を解消させようとしない。両方並べて、本人に関係を語らせる。

**時間差で同じ場所に着地させる：**

別の話題から入って、結果的に同じ場所に着くように聞く。
聞き手が矛盾を言語化しない。本人の口から出てくるように、場面を選ぶ。

**例外を聞く：**
> 「〇〇を大事にしてるって話だったんですが、それが通せなかった場面ってありますか？」

矛盾の指摘ではなく「例外の確認」。出てくるエピソードは Self Narrative との乖離そのもの。

**第三者の声を借りる：**
> 「周囲の人から『あの人は〇〇だ』って声があったりするんですが、自分的にはどうですか？」

攻撃の主体を聞き手から消す。

**身体に逃がす：**
> 「その話をしてる時、身体はどんな感じですか？」
> 「さっきの話とこっちの話、どっちの方が身体が楽ですか？」

言語を迂回して矛盾に触れる。身体感覚は嘘をつきにくい。

## 8. 聞き手の振る舞い

### やること

- 出てきた言葉を「入口」として扱い、その奥に進む
- 「今の言葉、自分的にしっくり来てますか？」で言語化の精度を本人に確認させる
- 言いよどみ、修正、沈黙を回答より重視する
- きれいにまとまった回答ほど、別角度からもう一度触る
- 相手の言葉をそのまま繰り返してから次の問いに入る
- 「なぜ」を3回聞く。表層で止めない
- 同じOSを複数の角度から確認する
- 沈黙を許容する。考え込んだ後に出てくる言葉を待つ
- 想定外の方向に話が行っても追う。本人が行きたい方向にエピソードがある
- 各セッション後にカバレッジを確認し、空白を埋めに行く
- 回答に含まれる Inference（本人の解釈）と Episode（事実）を聞き分ける
- 身体性に注意する。「その時、身体はどうだった？」を入れる

### やらないこと

- 「なるほど」「面白い」で評価しない。「もう少し聞いていいですか」で続ける
- 二択で回答を狭めない
- 聞き手が期待する方向を透かさない
- 数を指定しない（「3つ挙げて」は重みの順序を壊す）
- 深い質問を信頼構築の前にしない
- 気づかせようとしない（気づきは結果であって目的ではない）
- 本人の Inference をそのまま事実として扱わない
- 聞き手の仮説は会話中に出さない。記録時に仮説メモとして残す
- 「それは違うと思います」と本人の言葉を否定しない
- 本人の言語化を聞き手が上書きしない
- 言い直しを急かさない

## 9. フィードバックセッション

フィードバックの思想・設計は `concept.md` のフィードバック設計と活用設計を参照。
具体的な質問は `hearing-template.md` のフィードバックセクションを参照。

ここでは設計上の位置づけだけ記載する：

- 収束条件を満たした後に実施する
- Validation を通過した構造がある場合にのみ実施する
- 目的は「伝える」だけでなく「活用できる形にする」まで含む
- 本人の反応は新たな Episode として記録し、状態マップに反映する
